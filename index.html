Arquitectura Monolítica

Monolítica

Descripción

Aplicación desplegada como una unidad indivisible. Toda la lógica (presentación, negocio y datos) reside en un único proceso. Simple de desarrollar y operar en fases tempranas; costosa de escalar de forma granular.

Componentes Clave

  • UI Layer — renderizado / controladores MVC
  • Business Logic Layer — servicios y reglas de negocio
  • Data Access Layer — ORM / queries directas
  • Base de datos relacional única (PostgreSQL, MySQL)
  • Servidor de aplicación único (Tomcat, Gunicorn)

Flujo de Comunicación

El cliente realiza peticiones HTTP al servidor. Internamente, la solicitud transita en memoria entre capas (controlador → servicio → repositorio) sin red intermedia. El acceso a datos es una llamada directa al motor de base de datos.

Caso de Uso Típico

MVPs, herramientas internas, sistemas de bajo tráfico o equipos pequeños que priorizan velocidad de desarrollo sobre elasticidad.

Arquitectura de Microservicios

Microservicios

Descripción

Sistema compuesto por servicios pequeños, desplegables y escalables de forma independiente. Cada servicio encapsula una capacidad de negocio, posee su propia base de datos y expone una API bien definida.

Componentes Clave

  • API Gateway — punto de entrada único, enrutamiento y auth
  • Servicios de dominio — Users, Orders, Payments, Inventory
  • Message Broker — Kafka / RabbitMQ (comunicación async)
  • Service Registry — Consul / Eureka (descubrimiento)
  • Base de datos por servicio (polglot persistence)

Flujo de Comunicación

El API Gateway enruta peticiones síncronas (REST/gRPC) a los servicios. Las operaciones de larga duración usan eventos sobre un broker (pub/sub). Cada servicio lee y escribe exclusivamente su propia base de datos.

Caso de Uso Típico

Plataformas de e-commerce, fintechs y SaaS con equipos independientes, alta frecuencia de despliegue y necesidad de escalar dominios específicos.

Aplicación Web — 3-Tier / N-Tier

Web

Descripción

Separación física y lógica de la aplicación en capas horizontales: presentación, lógica de negocio y datos. N-Tier extiende el modelo añadiendo capas intermedias (caché, integración, seguridad) para mayor escalabilidad y separación de responsabilidades.

Componentes Clave

  • Tier 1 — CDN + Web Server (Nginx, static assets)
  • Tier 2 — App Server (REST API, lógica de negocio)
  • Tier 3 — Base de datos (RDBMS + réplicas lectura)
  • Caché distribuida — Redis / Memcached
  • Load Balancer — distribución horizontal de carga

Flujo de Comunicación

El cliente consume assets desde el CDN y envía peticiones al Load Balancer, que distribuye tráfico entre App Servers. La capa de aplicación consulta caché antes de acceder a la base de datos; escrituras van al nodo primario, lecturas a réplicas.

Caso de Uso Típico

Portales corporativos, e-commerce tradicional, sistemas ERP web y aplicaciones con patrones de tráfico predecibles y separación clara de equipos por capa.

Diagrama de Arquitectura

Aplicación Móvil — Cliente-Servidor / BFF

Mobile

Descripción

El patrón Backend for Frontend (BFF) introduce una capa intermediaria específica por canal (iOS, Android, Web) que agrega, transforma y optimiza datos de múltiples servicios backend, reduciendo latencia y over-fetching en dispositivos móviles.

Componentes Clave

  • Clientes nativos — iOS (Swift), Android (Kotlin)
  • BFF por plataforma — API tailored por canal
  • Auth Service — OAuth 2.0 / JWT
  • Servicios backend — dominio de negocio compartidos
  • Push Notification Gateway — APNs / FCM

Flujo de Comunicación

El cliente móvil se autentica vía OAuth y llama a su BFF dedicado. El BFF orquesta llamadas a múltiples servicios internos (REST/gRPC), agrega la respuesta y la devuelve optimizada para el formato y ancho de banda del canal móvil.

Caso de Uso Típico

Apps de banca móvil, marketplaces con app nativa y web, y cualquier producto con múltiples clientes que consumen backends compartidos con distintos requisitos de datos.

Diagrama de Arquitectura

Arquitectura Serverless

Serverless

Descripción

Ejecución de lógica de negocio en funciones efímeras (FaaS) gestionadas por el proveedor cloud. No requiere provisioning de servidores; la infraestructura escala automáticamente a cero. El modelo de costo es por invocación y tiempo de ejecución.

Componentes Clave

  • API Gateway — enrutamiento HTTP a funciones
  • Functions — AWS Lambda / Azure Functions / GCF
  • Event Sources — S3 triggers, SQS, EventBridge
  • Managed Storage — DynamoDB, S3, Firestore
  • IAM — permisos por función (least privilege)

Flujo de Comunicación

Un evento (HTTP, mensaje en cola, cambio en storage) invoca una función específica. La función se ejecuta en un contenedor efímero gestionado por el cloud provider, accede a servicios gestionados y termina. No existe estado en-proceso entre invocaciones.

Caso de Uso Típico

Procesamiento de eventos asíncronos, webhooks, ETL ligero, APIs con tráfico altamente variable y backends de bajo volumen que requieren coste operativo mínimo.

Arquitectura Basada en Contenedores

Docker / Kubernetes

Descripción

Empaquetado de aplicaciones en imágenes OCI ejecutadas como contenedores sobre un orquestador (Kubernetes). Permite portabilidad entre entornos, escalado declarativo, auto-healing y gestión de configuración estandarizada.

Componentes Clave

  • Container Registry — almacén de imágenes (ECR, GCR)
  • Kubernetes Control Plane — API Server, Scheduler, etcd
  • Worker Nodes — Pods con uno o más contenedores
  • Ingress Controller — enrutamiento HTTP externo
  • ConfigMaps / Secrets — gestión de configuración
  • Persistent Volumes — storage stateful (bases de datos)

Flujo de Comunicación

El tráfico externo llega al Ingress Controller, que enruta a Services (ClusterIP/LB). Los Services balancean carga entre Pods. El orquestador monitorea el estado deseado y reprograma Pods caídos automáticamente en nodos disponibles.

Caso de Uso Típico

Microservicios en producción, plataformas multi-tenant, pipelines CI/CD con entornos reproducibles y aplicaciones que requieren escalado horizontal automatizado y alta disponibilidad.
